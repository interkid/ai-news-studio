## M5: デザイン・ネタ・インパクト構成強化（次はここ）

実施順は「1本の動画の見た目品質」への効果が大きい順: **1 → 4 → 2 → 3**
（2026-07-06 見直し。4のプロンプトのビート構成テンプレ化は実装済み）
各項目の完了条件 = `studio render`でサンプル動画を出力し目視で品質確認すること。

**fal.aiの状態（2026-07-07更新）**: $10チャージ済み・疎通確認済みで**稼働中**。
GitHub Secretsにも`FAL_KEY`登録済み（CIのB-roll生成も有効）。

### 1. キャラ/背景素材の刷新（✅ キャラ完了 2026-07-07 / layoutの一部残り）
**キャラ構成の変更（ユーザ決定）**: きつね×うさぎは既存作品との類似感が強いため、
**「キャラメル色の垂れ耳解説うさぎ（丸眼鏡+赤蝶ネクタイ+紺ベスト）×白い聞き役うさぎ（黄スカーフ）」
のうさぎペア**へ変更。内部スピーカーID・ディレクトリ名は`fox`のまま（コード改修を最小化）。
- 画風はピクサー調3DCG（スタイル候補4案からユーザがAを選定）
- Nano Banana Pro編集モードで11状態を生成→rembgで透過→共通bboxで切り出し→コミット済み
- `mouth_closed`は`normal`のコピー（シルエット完全一致）
- `scripts/gen_character_assets.py`(生成) / `scripts/finalize_character_assets.py`(仕上げ)で再現可能
- `assets/layout/bg.png`も同画風（夜空+雲+ボケ光）へ差し替え済み
- **残り**: `logo.png`・`badge_*.png`・`broll_placeholder.png`の刷新（日本語テキストは
  AI生成が苦手なためPillow描画の色・形の改善で対応する方針）

**決定した方式（2026-07-06更新）**: グリッド一括生成ではなく、fal.ai経由の **Nano Banana Pro
（Gemini 3 Pro Image, $0.15/枚）** を第一候補に使う（キャラ同一性維持・表情差分が現行モデルで最も得意。
同じ`FAL_KEY`で呼べる。コスト重視の代替は FLUX.1 Kontext $0.04/枚）。
1. スタイル候補を3〜4案生成（例: ピクサー調3DCG / 毛並みまで描き込んだ厚塗り / リッチなアニメ調）し、
   人間が1案を選定。**キャラとしての質感・リアリティは高くてよい**（禁止は「実写写真と誤認される」
   写実=実在人物・実写風報道写真のみ。キャラと分かる3DCG/リッチイラストは制約に抵触しない）
2. 選定スタイルでベースキャラ（きつね・うさぎ各1枚）を確定
3. 編集モードで「同じキャラの表情違い・口の開閉違い」を差分生成（同一性の維持がグリッド方式より確実）
4. rembg等で背景を透過化 → `manifest.json`の各スロットへ配置
5. 品質不十分ならその時点で外部発注等を検討（予算制約に注意）

**素材仕様（`renderer/layout.py`・`manifest.json`から確定済み）**:
- 透過PNG。配置時に高さ760px(`CHAR_TARGET_H`)へ正規化・足元基準線揃え
- 表情違いでも**シルエット（ポーズ）は同一**にする（表情切替・8fps口パクでガタつくため顔のみ変える）
- 必要枚数: きつね6状態(normal/surprised/smug/worried/mouth_open/mouth_closed)、
  うさぎ5状態(normal/surprised/curious/mouth_open/mouth_closed)
- イラスト調のみ（写実禁止）・商用可ライセンス・`assets/LICENSES.md`に記録

あわせて `assets/layout/` も同じ画風で刷新する: `bg.png`（現状ほぼ無地グラデ）・`logo.png`・
コーナーバッジ4種(`badge_*.png`)・`broll_placeholder.png`。

### 4. フック/展開/オチのテンプレ化（`FAL_KEY`不要・並行して先に着手）
`scriptwriter/generate.py`のプロンプトに、exchanges全体のビート構成を明文化する:
- ①つかみ（きつね第一声。hookの言い直しではなく具体の一撃）
- ②リアクション+疑問（うさぎ）
- ③核心・数字・根拠（きつね）
- ④オチ/視聴者への示唆（「あなたの仕事なら〜」等、コメントしたくなる引き）

あわせて`scorer/`に「フック強度・オチの引き」の採点軸を追加し、2案からの選抜基準を
ビート構成と揃える。

### 2. 演出バリエーション追加（4の構成を映像で増幅する位置づけ）
`renderer/effects.py`（現状: zoom_in / pop_in / shake / slide_in / flash / ken_burns）を拡充。
TikTokは冒頭1秒が勝負なのに、現状のフック演出は0.5秒ズームインのみ。優先度の高い順:
- 冒頭インパクト演出（フラッシュ+シェイク+デカ文字ドロップの組み合わせ）
- 感情(surprised/smug等)連動のエフェクト自動選択
- その他バリエーション追加

### 3. ネタ選定への人間の事前レビュー機能（後回し。①〜③で動画品質を上げてから）
「ネタの質」の話で「1本の見た目品質」とは別軸、かつ3段ワークフロー化という最も重い改修のため
M5内では最後に実施。それまでは手動実行運用で人間の目は挟めている。
現状は`collector`のLLM採点のみで自動的に上位1件へ進む設計。これに人間の選定を挟む。

**決定した方式**: 既存のStage6承認（Cloudflare Worker + KV + `publish.yml`の15分ポーリング）と
同じパターンを踏襲する。
1. `daily.yml`の収集ステージ終了時点で候補トピック(上位3件など)をLINE Flexのボタン形式で通知し、
   台本生成前にジョブを終了する
2. 既存のCloudflare Worker/KVを流用し、`video:{id}`と同様に`topic:{run_id}`のようなキーへ
   選択結果を書き込む
3. 台本生成以降を担う新しいワークフロー（`publish.yml`と同様のポーリング方式、または既存構成の拡張）が
   選択済みトピックを検知し、Stage2以降を再開する

CIジョブを人間の返信待ちでブロックさせないため、「収集」と「生成」を別ジョブ・別トリガーに分割し、
現状の2段構成（daily→publish）を3段構成（収集→生成→承認/投稿）に拡張するイメージ。
